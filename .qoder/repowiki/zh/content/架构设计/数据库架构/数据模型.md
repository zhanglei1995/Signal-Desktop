## 引言
Signal-Desktop 的数据模型设计用于支持安全消息传递应用的核心功能，包括消息传递、会话管理、联系人存储和附件处理。该数据模型基于 SQLite 数据库实现，通过 TypeScript 类型定义和数据库迁移脚本确保数据结构的一致性和完整性。本文档详细描述了核心实体的结构、字段定义、约束关系以及数据验证规则。

## 核心实体结构

### 消息实体
消息实体（Message）是 Signal-Desktop 数据模型中最核心的组成部分，用于存储用户发送和接收的所有消息内容。消息实体的结构在 `model-types.d.ts` 文件中通过 `MessageAttributesType` 类型定义。

**字段定义**:
- **id**: 消息的唯一标识符，字符串类型，主键
- **type**: 消息类型，枚举值包括 'incoming', 'outgoing', 'call-history' 等
- **body**: 消息正文内容，字符串类型，可为空
- **attachments**: 附件数组，包含图片、视频、文件等多媒体内容
- **conversationId**: 关联的会话ID，字符串类型，外键
- **sent_at**: 消息发送时间戳，数字类型
- **received_at**: 消息接收时间戳，数字类型，用于会话排序
- **readStatus**: 阅读状态，枚举值包括 'unread', 'read', 'viewed'
- **expireTimer**: 消息自动销毁计时器，以秒为单位
- **quote**: 引用消息，包含被引用消息的内容和作者信息
- **reactions**: 消息反应（表情回复）数组
- **preview**: 链接预览信息数组
- **sticker**: 表情贴纸信息
- **contact**: 嵌入的联系人信息数组
- **poll**: 投票消息属性
- **schemaVersion**: 消息模式版本，用于数据迁移和兼容性处理

**数据类型与约束**:
- 所有时间戳字段均为数字类型，表示 Unix 时间戳（毫秒）
- `id` 字段为必填项，作为主键确保唯一性
- `conversationId` 字段为必填项，建立与会话实体的外键关系
- `sent_at` 和 `received_at` 字段为必填项，用于消息排序和时间线展示
- 附件相关字段通过 `attachments` 数组统一管理，每个附件对象包含路径、大小、内容类型等元数据

### 会话实体
会话实体（Conversation）代表用户与单个联系人或群组的对话关系。会话实体在 `model-types.d.ts` 文件中通过 `ConversationAttributesType` 类型定义。

**字段定义**:
- **id**: 会话的唯一标识符，字符串类型，主键
- **type**: 会话类型，'private' 或 'group'
- **name**: 会话名称，字符串类型，可为空
- **e164**: 联系人电话号码，字符串类型，E.164 格式
- **serviceId**: 服务ID，字符串类型，用于标识用户
- **groupId**: 群组ID，字符串类型，仅群组会话使用
- **active_at**: 最后活跃时间戳，数字类型
- **lastMessage**: 最后一条消息内容摘要，字符串类型
- **unreadCount**: 未读消息计数，数字类型
- **isArchived**: 是否已归档，布尔类型
- **isPinned**: 是否置顶，布尔类型
- **expireTimer**: 默认消息销毁计时器，以秒为单位
- **members**: 群组成员ID数组，仅群组会话使用
- **membersV2**: V2 群组成员信息数组，包含 ACI、角色等详细信息
- **avatar**: 会话头像信息，包含路径和元数据
- **color**: 会话颜色主题，字符串类型
- **profileSharing**: 是否共享个人资料，布尔类型
- **storageID**: 存储服务ID，用于云同步
- **version**: 数据版本号，用于同步冲突解决

**数据类型与约束**:
- `id` 字段为必填项，作为主键确保唯一性
- `type` 字段为必填项，限定为 'private' 或 'group'
- 私人会话必须包含 `e164` 或 `serviceId` 字段
- 群组会话必须包含 `groupId` 字段
- `active_at` 字段用于会话列表排序
- `unreadCount` 字段维护未读消息数量，支持快速状态展示
- 群组成员信息通过 `membersV2` 数组详细记录，包含成员角色、加入时间等元数据

### 联系人实体
联系人实体实际上是会话实体的一个特例，当会话类型为 'private' 时，即表示与单个联系人的对话。联系人相关信息主要存储在会话实体中，通过特定字段表示联系人属性。

**关键字段**:
- **profileName**: 联系人显示名称，字符串类型
- **profileFamilyName**: 联系人姓氏，字符串类型
- **about**: 联系人简介，字符串类型
- **aboutEmoji**: 联系人简介表情符号，字符串类型
- **profileKey**: 联系人资料密钥，用于端到端加密的资料验证
- **verified**: 验证状态，数字类型，表示联系人身份验证级别
- **profileLastUpdatedAt**: 资料最后更新时间，数字类型
- **profileAvatar**: 联系人头像信息，包含路径和元数据
- **username**: 用户名，字符串类型，可选

**特殊属性**:
- 联系人资料通过 `profileKey` 实现端到端加密验证，确保资料未被篡改
- `verified` 状态通过安全号码验证流程更新，提供身份确认保障
- 联系人头像和名称信息支持本地缓存和服务器同步
- 资料更新时间戳用于优化同步策略，减少不必要的数据传输

### 附件实体
附件实体用于管理消息中的多媒体内容和文件。与传统数据库设计不同，Signal-Desktop 将附件信息作为消息的组成部分进行管理，而不是独立的实体表。

**附件结构**:
附件信息嵌入在消息实体的 `attachments` 数组中，每个附件对象包含以下字段：
- **contentType**: 内容类型，MIME 类型字符串，如 'image/jpeg', 'video/mp4'
- **fileName**: 文件名，字符串类型，可为空
- **size**: 文件大小，数字类型，字节为单位
- **width**: 图像宽度，数字类型，仅图像附件
- **height**: 图像高度，数字类型，仅图像附件
- **path**: 本地文件路径，字符串类型
- **thumbnail**: 缩略图信息，包含缩略图的路径和元数据
- **flags**: 附件标志位，数字类型，用于标记特殊属性
- **key**: 加密密钥，字符串类型，用于端到端加密
- **digest**: 内容摘要，字符串类型，用于完整性验证
- **blurHash**: 图像模糊哈希，字符串类型，用于快速预览
- **caption**: 附件说明文字，字符串类型

**存储策略**:
- 附件文件存储在本地文件系统中，通过 `path` 字段引用
- 所有附件内容在传输和存储时均进行端到端加密
- 缩略图用于消息列表中的快速预览，提升用户体验
- 大文件支持分块传输和渐进式加载
- 附件元数据与消息内容一起同步，确保一致性

## 数据库模式与关系

### 数据库表结构
Signal-Desktop 使用 SQLite 作为本地数据库，通过一系列迁移脚本管理数据库模式的演进。核心表结构包括：

**messages 表**:
- 主键: `id`
- 核心字段: `json` (存储完整消息对象的 JSON 序列化), `conversationId`, `received_at`, `sent_at`
- 索引: `received_at` (用于时间线排序), `conversationId` (用于会话过滤)

**conversations 表**:
- 主键: `id`
- 核心字段: `json` (存储完整会话对象的 JSON 序列化), `active_at`, `type`
- 索引: `active_at` (用于会话列表排序), `type` (用于过滤私人/群组会话)

**message_attachments 表**:
- 复合主键: `(messageId, editHistoryIndex, attachmentType, orderInMessage)`
- 外键: `messageId` REFERENCES `messages(id)` ON DELETE CASCADE
- 核心字段: 各种附件元数据字段
- 索引: 多个复合索引用于优化附件查询性能

**其他重要表**:
- `notificationProfiles`: 通知配置文件表，存储用户自定义的通知规则
- `chatFolders`: 聊天文件夹表，用于组织和分类会话
- `callLinks`: 通话链接表，存储群组通话的链接信息
- `stickerPacks`: 表情包表，管理用户安装的表情包

### 索引策略
数据库索引设计旨在优化常见查询模式的性能：

**消息查询索引**:
- `messages_received_at_index`: 基于 `received_at` 字段的索引，用于快速获取时间线消息
- `messages_conversationId_received_at_index`: 基于 `conversationId` 和 `received_at` 的复合索引，用于获取特定会话的消息
- `messages_unread_index`: 基于阅读状态的索引，用于快速统计未读消息

**会话查询索引**:
- `conversations_active_at_index`: 基于 `active_at` 字段的索引，用于会话列表排序
- `conversations_type_active_at_index`: 基于 `type` 和 `active_at` 的复合索引，用于分类会话列表
- `conversations_unread_count_index`: 基于未读计数的索引，用于突出显示重要会话

**附件查询索引**:
- `message_attachments_messageId_index`: 基于 `messageId` 的索引，用于获取消息的所有附件
- `message_attachments_plaintextHash_index`: 基于内容哈希的索引，用于去重检测
- `message_attachments_path_index`: 基于文件路径的索引，用于文件系统清理

## 数据验证与业务逻辑

### 数据完整性约束
Signal-Desktop 通过多层次的验证机制确保数据完整性：

**主键约束**:
- 所有实体必须具有唯一标识符（`id` 字段）
- 消息ID使用 UUID 保证全局唯一性
- 会话ID在本地生成，确保不会冲突

**外键约束**:
- 消息必须关联到有效的会话
- 附件必须关联到有效的消息
- 引用消息必须指向存在的消息记录
- 删除会话时级联删除所有相关消息

**数据类型约束**:
- 时间戳字段必须为有效数字
- 布尔字段必须为 0 或 1（SQLite 中的布尔表示）
- 枚举字段必须符合预定义的值集合
- JSON 字段必须包含有效的 JSON 数据

**业务规则约束**:
- 私人会话不能有多个成员
- 群组会话必须有至少一个管理员
- 消息销毁计时器必须在合理范围内（0-4周）
- 会话名称不能超过特定长度限制
- 附件大小不能超过系统限制

### 业务逻辑约束
核心业务逻辑通过代码实现，确保数据的一致性和安全性：

**消息发送逻辑**:
- 发送消息前验证会话状态和网络连接
- 为每条消息生成唯一ID和时间戳
- 加密消息内容和附件
- 更新会话的最后活跃时间和最后消息摘要
- 维护未读计数和阅读状态

**会话管理逻辑**:
- 创建新会话时初始化默认属性
- 更新会话成员时验证权限和角色
- 处理群组邀请和成员加入/退出
- 维护会话的归档状态和置顶状态
- 同步会话设置到存储服务

**附件处理逻辑**:
- 上传附件前进行压缩和格式转换
- 为附件生成缩略图和模糊哈希
- 加密附件内容并生成完整性校验
- 管理附件的下载状态和错误重试
- 清理不再使用的附件文件

**数据同步逻辑**:
- 生成变更日志用于增量同步
- 处理同步冲突和版本冲突
- 验证同步数据的完整性和一致性
- 维护本地和远程数据的一致性
- 处理离线状态下的数据同步
