## 简介
Signal-Desktop的持久化配置系统采用分层架构，通过用户配置(user_config)和默认配置(default.json)的合并机制实现灵活的配置管理。系统在启动时根据环境变量加载相应的配置文件，并将用户自定义设置持久化到本地存储中。

## 配置文件结构
Signal-Desktop的配置系统由多个JSON文件组成，形成一个层次化的配置结构。核心配置文件包括：

- **default.json**: 包含所有配置项的默认值，适用于所有环境
- **production.json**: 生产环境特定的配置覆盖
- **development.json**: 开发环境特定的配置覆盖
- **staging.json**: 预发布环境特定的配置覆盖

配置项采用嵌套的JSON对象结构，包含服务器URL、CDN配置、安全参数等关键信息。

## 序列化与反序列化
配置系统的序列化与反序列化过程通过`base_config.node.ts`中的`start`函数实现，确保配置数据在内存和磁盘之间的正确转换。

```mermaid
sequenceDiagram
participant Memory as 内存缓存
participant Disk as 磁盘文件
participant BaseConfig as BaseConfig模块
BaseConfig->>Disk : readFileSync(targetPath)
Disk-->>BaseConfig : 返回JSON字符串
BaseConfig->>BaseConfig : JSON.parse(字符串)
BaseConfig->>Memory : 存储到cachedValue
BaseConfig->>Memory : 初始化空对象(如文件不存在)
loop 配置更新
BaseConfig->>Memory : set(keyPath, value)
BaseConfig->>BaseConfig : 创建新配置对象
BaseConfig->>BaseConfig : JSON.stringify(新对象, null, ' ')
BaseConfig->>Disk : writeFileSync(targetPath, JSON字符串)
Disk-->>BaseConfig : 写入成功/失败
alt 写入成功
BaseConfig->>Memory : 更新cachedValue
else 写入失败
BaseConfig->>Console : 记录警告日志
BaseConfig->>Memory : 仅更新内存数据
end
end
```

## 迁移与升级策略
系统采用渐进式的配置迁移策略，确保数据在版本升级过程中的完整性和一致性。

```mermaid
flowchart TD
Start["版本升级"] --> BackupConfig["备份当前配置"]
BackupConfig --> AnalyzeChanges["分析配置变更"]
AnalyzeChanges --> |新增配置项| AddDefaults["添加默认值"]
AnalyzeChanges --> |移除配置项| RemoveObsolete["移除过时项"]
AnalyzeChanges --> |修改配置项| TransformData["转换数据格式"]
AddDefaults --> ValidateNewConfig["验证新配置"]
RemoveObsolete --> ValidateNewConfig
TransformData --> ValidateNewConfig
ValidateNewConfig --> |验证失败| RevertChanges["回滚变更"]
ValidateNewConfig --> |验证成功| SaveNewConfig["保存新配置"]
SaveNewConfig --> UpdateVersion["更新版本号"]
UpdateVersion --> Cleanup["清理临时文件"]
Cleanup --> End["迁移完成"]
RevertChanges --> RestoreBackup["恢复备份"]
RestoreBackup --> End
```
